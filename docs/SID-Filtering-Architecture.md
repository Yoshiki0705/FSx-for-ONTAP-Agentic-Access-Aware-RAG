# SIDベース権限フィルタリング アーキテクチャ

## 概要

本システムは、NTFS ACLのSID（Security Identifier）を活用して、RAG検索結果をユーザーごとにフィルタリングします。FSx for NetApp ONTAPのファイルシステム上のアクセス権限情報を、ベクトルデータベースのメタデータとして保持し、検索時にリアルタイムで権限チェックを行います。

## SIDフィルタリングの詳細ロジック

### Step 1: ユーザーSIDの取得

ユーザーがチャットで質問を送信すると、KB Retrieve APIがDynamoDB `user-access` テーブルからユーザーのSID情報を取得します。

```
DynamoDB user-access テーブル
┌──────────────────────────────────────────────────────────────┐
│ userId (PK)          │ userSID              │ groupSIDs      │
├──────────────────────┼──────────────────────┼────────────────┤
│ {Cognito sub}        │ S-1-5-21-...-500     │ [S-1-5-21-...-512, │
│ (admin@example.com)  │ (Administrator)      │  S-1-1-0]      │
├──────────────────────┼──────────────────────┼────────────────┤
│ {Cognito sub}        │ S-1-5-21-...-1001    │ [S-1-1-0]      │
│ (user@example.com)   │ (Regular User)       │                │
└──────────────────────┴──────────────────────┴────────────────┘

→ ユーザーの全SIDリスト = [userSID] + groupSIDs
   admin: [S-1-5-21-...-500, S-1-5-21-...-512, S-1-1-0]
   user:  [S-1-5-21-...-1001, S-1-1-0]
```

### Step 2: ドキュメントメタデータの取得

Bedrock KB検索結果の各Citationには、S3上の `.metadata.json` から取り込まれたメタデータが含まれます。

> **`.metadata.json`の作成方法**: 本システムにはAD同期Lambda（`lambda/agent-core-ad-sync/`）とFSx権限サービス（`lambda/permissions/fsx-permission-service.ts`）によるNTFS ACL自動取得が実装されています。デモ環境では検証用にサンプルデータを手動配置しています。詳細は [docs/embedding-server-design.md](embedding-server-design.md) の「メタデータの構造」セクションを参照してください。

```
ドキュメントメタデータ（.metadata.json）
┌──────────────────────────┬──────────────────────────────────────┐
│ ドキュメント              │ allowed_group_sids                   │
├──────────────────────────┼──────────────────────────────────────┤
│ public/product-catalog   │ ["S-1-1-0"]                          │
│                          │  └─ Everyone（全ユーザー）            │
├──────────────────────────┼──────────────────────────────────────┤
│ public/company-overview  │ ["S-1-1-0"]                          │
│                          │  └─ Everyone（全ユーザー）            │
├──────────────────────────┼──────────────────────────────────────┤
│ confidential/financial   │ ["S-1-5-21-...-512"]                 │
│                          │  └─ Domain Admins のみ               │
├──────────────────────────┼──────────────────────────────────────┤
│ confidential/hr-policy   │ ["S-1-5-21-...-512"]                 │
│                          │  └─ Domain Admins のみ               │
├──────────────────────────┼──────────────────────────────────────┤
│ restricted/project-plan  │ ["S-1-5-21-...-1100",                │
│                          │  "S-1-5-21-...-512"]                 │
│                          │  └─ Engineering + Domain Admins      │
└──────────────────────────┴──────────────────────────────────────┘
```

### Step 3: SIDマッチング

ユーザーのSIDリストとドキュメントの `allowed_group_sids` を照合します。

```
マッチング判定: ユーザーSID ∩ ドキュメントSID ≠ ∅ → ALLOW

■ 管理者ユーザー (admin@example.com)
  ユーザーSID: [S-1-5-21-...-500, S-1-5-21-...-512, S-1-1-0]

  public/product-catalog    → S-1-1-0 ∈ ユーザーSID → ✅ ALLOW
  public/company-overview   → S-1-1-0 ∈ ユーザーSID → ✅ ALLOW
  confidential/financial    → S-1-5-21-...-512 ∈ ユーザーSID → ✅ ALLOW
  confidential/hr-policy    → S-1-5-21-...-512 ∈ ユーザーSID → ✅ ALLOW
  restricted/project-plan   → S-1-5-21-...-512 ∈ ユーザーSID → ✅ ALLOW

■ 一般ユーザー (user@example.com)
  ユーザーSID: [S-1-5-21-...-1001, S-1-1-0]

  public/product-catalog    → S-1-1-0 ∈ ユーザーSID → ✅ ALLOW
  public/company-overview   → S-1-1-0 ∈ ユーザーSID → ✅ ALLOW
  confidential/financial    → S-1-5-21-...-512 ∉ ユーザーSID → ❌ DENY
  confidential/hr-policy    → S-1-5-21-...-512 ∉ ユーザーSID → ❌ DENY
  restricted/project-plan   → {-1100, -512} ∩ {-1001, S-1-1-0} = ∅ → ❌ DENY
```

### Step 4: 安全側フォールバック

SID情報が取得できない場合（DynamoDBにレコードなし、接続エラー等）は、安全側にフォールバックして全ドキュメントへのアクセスを拒否します。

```
SID取得失敗時のフロー:
  DynamoDB → エラー or レコードなし
    → allUserSIDs = [] (空)
    → 全ドキュメント DENY
    → filterMethod: "DENY_ALL_FALLBACK"
```

## SID（Security Identifier）について

### SIDの構造

```
S-1-5-21-{ドメインID1}-{ドメインID2}-{ドメインID3}-{RID}
│ │ │  │  └─────────────────────────────────────────┘  └─┘
│ │ │  │              ドメイン識別子                    相対ID
│ │ │  └─ サブオーソリティ数
│ │ └─ 識別オーソリティ（5 = NT Authority）
│ └─ リビジョン
└─ SIDプレフィックス
```

### 主要なSID

| SID | 名前 | 説明 |
|-----|------|------|
| `S-1-1-0` | Everyone | すべてのユーザー |
| `S-1-5-21-...-500` | Administrator | ドメイン管理者 |
| `S-1-5-21-...-512` | Domain Admins | ドメイン管理者グループ |
| `S-1-5-21-...-1001` | User | 一般ユーザー |
| `S-1-5-21-...-1100` | Engineering | エンジニアリンググループ |

### FSx for ONTAP における SID

FSx for ONTAPはNTFSセキュリティスタイルのボリュームでWindows ACLをサポートします。各ファイル/ディレクトリにはACL（Access Control List）が設定され、SIDベースでアクセス権限が管理されます。

S3 Access Pointを通じてFSx上のファイルにアクセスする場合、NTFS ACL情報がメタデータとして公開されます。本システムでは、このACL情報（SID）をBedrock KBのメタデータとして取り込み、検索時のフィルタリングに活用しています。

### 1. データ取り込み時（Embedding）

```
FSx for ONTAP                    S3 Access Point              Bedrock KB

│  Users:Read │                │ file.md      │             │              │
│             │                │ file.md      │             └──────┬───────┘
│ file.md     │                │ .metadata    │                    │
│ .metadata   │                │ .json        │                    ▼
```

> S3 Access PointはFSxボリュームのファイルを直接S3互換で公開するため、S3バケットへのコピーは不要です。

```

  │ 質問送信         │                    │                  │                │
  │─────────────────▶│                    │                  │                │

  │                  │ userSID + groupSIDs│                  │                │
  │                  │                    │                  │                │
  │                  │ Retrieve API（ベクトル検索+メタデータ） │                │
  │                  │─────────────────────────────────────▶│                │
  │                  │◀─────────────────────────────────────│                │
  │                  │ 検索結果 + メタデータ(SID)            │                │
  │                  │                    │                  │                │
  │                  │ SIDマッチング      │                  │                │
  │                  │ (ユーザーSID ∩     │                  │                │
  │                  │  ドキュメントSID)  │                  │                │
  │                  │                    │                  │                │
  │                  │ 許可ドキュメントのみで回答生成                         │
  │                  │──────────────────────────────────────────────────────▶│
  │                  │◀──────────────────────────────────────────────────────│

  │◀─────────────────│                    │                  │                │
```

> RetrieveAndGenerate APIではなくRetrieve APIを使用する理由: RetrieveAndGenerate APIはcitationの`metadata`フィールドに`.metadata.json`の`allowed_group_sids`を含めないため、SIDフィルタリングが不可能です。Retrieve APIはメタデータを正しく返すため、2段階方式（Retrieve → SIDフィルタ → Converse）を採用しています。

### 3. Agentモード検索時（ハイブリッド方式）

Agentモードでは、Permission-awareなRAGを実現するためにハイブリッド方式を採用しています。InvokeAgent APIはアプリ側でのSIDフィルタリングの余地がないため、KB Retrieve API + SIDフィルタリング + Converse API（Agent用システムプロンプト付き）の組み合わせで実現します。

```
ユーザー          Next.js API           Bedrock KB          DynamoDB         Converse API
  │                  │                    │                    │                │
  │ 質問送信         │                    │                    │                │
  │─────────────────▶│                    │                    │                │
  │                  │ Retrieve API       │                    │                │
  │                  │───────────────────▶│                    │                │
  │                  │◀───────────────────│                    │                │
  │                  │ 検索結果+メタデータ │                    │                │
  │                  │                    │                    │                │
  │                  │ SID取得                                 │                │
  │                  │────────────────────────────────────────▶│                │
  │                  │◀────────────────────────────────────────│                │
  │                  │                    │                    │                │
  │                  │ SIDフィルタリング  │                    │                │
  │                  │ (KBモードと同じ)   │                    │                │
  │                  │                    │                    │                │
  │                  │ 許可ドキュメント + Agent用システムプロンプトで回答生成    │
  │                  │─────────────────────────────────────────────────────────▶│
  │                  │◀─────────────────────────────────────────────────────────│
  │                  │                    │                    │                │
  │ Agent応答+Citation│                    │                    │                │
  │◀─────────────────│                    │                    │                │
```

> Bedrock Agent InvokeAgent APIも利用可能ですが、InvokeAgent APIはアプリ側でのSIDフィルタリングの余地がないため、フォールバックとしてのみ使用されます。Permission-awareを保証するため、ハイブリッド方式がデフォルトです。

## セキュリティ設計

### 安全側フォールバック原則

本システムは「Fail-Closed」の原則に従い、権限チェックに失敗した場合は全ドキュメントへのアクセスを拒否します。

| 状況 | 動作 |
|------|------|
| DynamoDB接続エラー | 全ドキュメント拒否 |
| ユーザーSIDレコードなし | 全ドキュメント拒否 |
| メタデータにSID情報なし | 該当ドキュメント拒否 |
| SIDマッチなし | 該当ドキュメント拒否 |
| SIDマッチあり | 該当ドキュメント許可 |

### 権限キャッシュ

DynamoDB `permission-cache` テーブルにフィルタリング結果をキャッシュし、同一ユーザー・同一ドキュメントの繰り返しチェックを高速化します（TTL: 5分）。
